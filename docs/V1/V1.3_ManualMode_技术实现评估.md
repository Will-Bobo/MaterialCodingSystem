> 目标：在**不删除 Auto 自动编码能力**、且 UI 默认仅开放 Manual 的前提下，把当前 PRD 中新增/收敛的规则落实到现有代码结构（SQLite + Dapper + UnitOfWork + ApplicationService）。

> 评估基线：以当前实现文件为准（示例：`MaterialCodingSystem.Infrastructure.Sqlite.SqliteSchema`、`SqliteMaterialRepository`、`MaterialApplicationService`）。

## 1. 当前代码实现概览（与 PRD 的映射）

## 2. 与最新 PRD（Manual 补充规则 + spec 启用态唯一）的差距与冲突

### 2.1 spec 唯一性（启用态唯一）与当前 Schema/查询的冲突（必须改）

PRD 口径：`UNIQUE(category_code, spec) WHERE status = 1`（仅启用态唯一，废弃可复用 spec）

当前代码现状：

- Schema：`UNIQUE(category_code, spec)`（会导致 **status=0 仍占用 spec**，不允许复用）
- Repository：`SpecExistsAsync` 查询 **未过滤 `status=1`**

依据代码：

```119:125:d:\wpf_workspace\MaterialCodingSystem\MaterialCodingSystem\Infrastructure\Sqlite\SqliteMaterialRepository.cs
var sql = "SELECT 1 FROM material_item WHERE category_code = @categoryCode AND spec = @spec LIMIT 1;";
```

影响：

- 违反 PRD 的“废弃记录不占用 spec，可重新新建物料”。
- 即使 Application 改成“只查 status=1”，DB 仍会因 `UNIQUE(category_code, spec)` 阻止插入（根因在 DDL）。

### 2.2 Manual code 规则当前未落地（需要新增/改造）

PRD Manual 要求：

- 输入标准化：`trim + ToUpperInvariant`
- 格式正则：`^[A-Z]{3}[0-9]{7}[A-Z]$`
- 解析：`category_code / serial_no / suffix`
  - `serial_no >= 1`（`0000000` -> `CODE_FORMAT_INVALID`）
  - 新建主物料 A：`suffix` 必须为 `A`（否则 `SUFFIX_INVALID`）
- 分类一致性：解析出来的 category_code 必须等于表单选择（否则 `CATEGORY_MISMATCH`）
- Group 规则：按 `(category_id, serial_no)` 查 group，存在复用，不存在创建；**禁止** max+1
- 并发：插入 group 命中 `UNIQUE(category_id, serial_no)` → 重新查询并复用，不算失败

当前代码尚无：

- `CreateMaterialItemA` 的 Manual 分支或独立 API
- `code` 解析/正则校验的 Domain ValueObject
- 按 `(category_id, serial_no)` 查询 group 的 Repository 方法
- group 并发“插入失败 -> 查询复用”的处理逻辑

## 3. 推荐的最小改造方案（按层落地）

> 原则：保持现有 Clean-ish 分层结构（Application 调度、Repository 纯 SQL），只做必要新增；Auto 逻辑不回归、不删。

### 3.1 Domain 层（建议新增 ValueObject）

新增 `ManualMaterialCode`（或 `ParsedMaterialCode`）：

- 输入：raw code string
- 标准化：`trim + upper`
- 校验正则
- 解析：`CategoryCode` + `SerialNo(int)` + `Suffix(char)`
- 规则：
  - `serial_no >= 1`，否则抛 `DomainException("CODE_FORMAT_INVALID")`
  - suffix 非 A-Z：`SUFFIX_INVALID`

好处：

- 校验语义集中、可测试
- Application 只做“组装流程”和错误码映射

### 3.2 Application 层（新增 Manual 创建入口，避免污染 Auto）

新增用例（推荐独立方法，避免在 `CreateMaterialItemA` 内塞分支导致语义混淆）：

- `CreateMaterialItemA_Manual(CreateMaterialItemAManualRequest)`
  - 入参：`category_code`（表单选择）+ `code`（用户输入）+ `spec/description/brand`
  - 流程按 PRD 15.1.1.1（你们刚写进 PRD 的那段）执行
  - 错误码：
    - `CODE_FORMAT_INVALID` / `CATEGORY_MISMATCH` / `SUFFIX_INVALID`
    - `CODE_DUPLICATE` / `SPEC_DUPLICATE`
    - `CATEGORY_NOT_FOUND` / `VALIDATION_ERROR`

同时保留现有 Auto：

- `CreateMaterialItemA` 继续走 max+1
- UI 默认只调用 Manual 新入口（Auto 入口隐藏）

### 3.3 Infrastructure（Repository）层：补齐“按 serial_no 查/插 group”的能力

需要新增接口与实现（示意）：

- `Task<int?> GetGroupIdByCategoryAndSerialNoAsync(CategoryCode categoryCode, int serialNo, ct)`
  - SQL：`SELECT id FROM material_group WHERE category_code=@categoryCode AND serial_no=@serialNo LIMIT 1`
  - 说明：当前表里同时存了 `category_id` 与 `category_code`，用 `category_code` 查能避免额外 join；但最终一致性仍以 `category_id` 为准。

- `InsertGroupAsync(categoryCode, serialNo)` 已存在，可复用

并发复用策略（Manual）：

- 先查 group：
  - 找到 -> 复用 id
  - 找不到 -> 尝试 InsertGroupAsync
    - 若命中 `CONSTRAINT_GROUP_CATEGORY_SERIAL` -> 再查一次并复用（**不抛出业务失败**）

### 3.4 数据库（Schema）必须调整：spec 启用态唯一

**当前 `UNIQUE(category_code, spec)` 必须移除**，替换为部分唯一索引：

- SQLite 支持：`CREATE UNIQUE INDEX ... WHERE status=1`

建议迁移步骤（可在 EnsureCreated 后增加“幂等迁移”逻辑，或引入显式 migration）：

1. 检测旧唯一索引/约束存在（SQLite 对表级 UNIQUE 约束不可直接 DROP，需要重建表或迁移数据）
2. 推荐最稳方案：**新建表 + 拷贝数据 + 重命名**（一次性 migration）
3. 建立：
   - `CREATE UNIQUE INDEX ux_material_item_category_spec_active ON material_item(category_code, spec) WHERE status=1;`
   - 以及 `(category_code, spec, status)` 或 `(category_code, spec)` 普通索引可选

风险提示：

- 若线上已存在违反“启用态唯一”的脏数据，需要先清洗，否则建索引会失败。

## 4. 并发语义评估（与当前重试框架的匹配）

### 4.1 Auto（现状）

`CreateMaterialItemA` 外层 `ExecuteWithRetry` 仅对 `CONSTRAINT_GROUP_CATEGORY_SERIAL` 做事务级重试，符合“max+1 并发冲突重试”的语义。

### 4.2 Manual（PRD 新增）

Manual 的 group 并发规则是：

- 插入 group 命中 `UNIQUE(category_id, serial_no)` → **转为复用**（重新查询 group）
- 不需要事务级重试（因为 serial_no 是用户指定，不存在“再分配”的空间）

因此 Manual 的策略应当是：

- **不复用 Auto 的 retryConstraint=CONSTRAINT_GROUP_CATEGORY_SERIAL** 语义
- 将该约束冲突当作“并发创建同一 group 的正常竞态”，转为查询复用

## 6. 测试建议（覆盖最关键风险）

- **Manual code 标准化**
  - `" elc0000123a "` → 标准化后通过，解析正确
- **serial_no 范围**
  - `ELC0000000A` → `CODE_FORMAT_INVALID`
- **分类不一致**
  - 选择 `ELC`，输入 `ABC0000123A` → `CATEGORY_MISMATCH`
- **suffix 不是 A**
  - `ELC0000123B`（新建主物料）→ `SUFFIX_INVALID`
- **spec 启用态唯一**
  - 同分类已有 status=1 同 spec → `SPEC_DUPLICATE`
  - 仅存在 status=0 同 spec → 允许新建
- **Manual group 并发复用**
  - 两个并发请求同 `ELC0000123A`：一个插入 group 成功，另一个命中唯一冲突后能查到并复用，最终都能插入 item（但第二个应因 `UNIQUE(code)` 失败 → `CODE_DUPLICATE`，这是符合预期的：同一 code 不能重复创建）

---

## 7. 结论（可行性与风险）

- **可行性**：现有架构（UoW + Repository + ApplicationService + Domain）支持以较小改造实现 Manual 模式；Auto 逻辑可保持不动。
- **关键风险点**：
  1) spec 启用态唯一需要 **真实的 DDL 迁移**（不能只改查询）
  2) Manual 与 Auto 的并发语义不同：Auto 需要重试，Manual 需要“冲突转复用”，不要混用同一套 retry 策略
  3) 现有 UI/请求 DTO 需要新增 `code` 字段并指向 Manual API（Auto 入口隐藏）
