# V1.4 BOM审核中心（轻量归档版）技术实施方案

> 目标：在 **不破坏 V1.3 冻结功能**（手工建料、替代料、搜索、导出、数据库备份恢复）的前提下，落地 PRD `7.7 BOM审核中心（V1.4）` 及其增量修订：**不启用 BOM 场景自动编码建料**，`code` 为空必为异常，仅展示；审核通过后 **归档保存原始 Excel 文件**，不拆明细入库，仅新增轻量索引表 `bom_archive`。
>
> 本文仅为技术实施方案，不生成代码。

---

## 1. 范围与约束（来自 PRD 的冻结口径）

### 1.1 必须实现（V1.4）

- **BOM Excel 导入**：用户选择本地 Excel 文件上传（客户端本地读取）。
- **自动审核**：解析表头与明细行，对每行输出审核状态：`PASS / NEW / ERROR`，并输出 **异常原因**。
- **新料导入**：仅对 `NEW` 行允许导入物料库（支持单行导入与一键导入全部新料）。
- **归档保存**：仅当 `NEW=0` 且 `ERROR=0` 时允许“保存审核通过BOM”；保存用户上传的 **原始 Excel 文件**，保持原封不动。
- **历史查询（轻量索引）**：通过 `bom_archive` 查询归档历史与定位文件路径。

### 1.2 明确禁止（V1.4）

（必须写死到实现与测试中）

- **禁止**根据空编码（`code` 为空）自动生成新编码建料。
- **禁止**在 BOM 页面内手工输入/编辑编码修复异常行。
- **禁止**自动修改 BOM 原文件内容（包括另存为重写、改格式、补列等）。
- **不做**：数量校验、位号校验、品牌标准化、用量分析、差异比对。

## 2. 总体架构落点（Clean Architecture + MVVM）

### 2.1 分层职责

- **Presentation（WPF/MVVM）**
  - 负责文件选择、状态展示、命令触发与错误提示。
  - 不做 Excel 解析、不做审核规则、不做入库与文件 IO 编排。
- **Application（Use Case）**
  - 负责用例编排：解析→审核→导入→归档→写索引。
  - 控制事务边界与错误码映射；对 UI 返回稳定 DTO。
- **Domain（规则）**
  - 定义 BOM 审核的纯规则（状态判定、异常原因枚举/码）。
  - 定义归档相关业务规则与前置条件（仅规则，不做任何环境/路径/IO职责），包括：
    - 是否允许归档（`NEW=0 且 ERROR=0`）
    - 版本是否重复（同 `finished_code + version` 不允许覆盖）
    - 归档行为的业务前置条件（例如：必须已完成审核、必须无 NEW/ERROR）
- **Infrastructure（实现）**
  - Excel 读取（使用现有 ClosedXML 读取，不写回）。
  - 文件归档（创建目录、文件 copy、临时文件重命名、文件存在性检查、清理临时文件）。
  - SQLite 仓储（`bom_archive` + 物料库既有表）。

补充（避免分层错误，必须一致）：

- **目标路径生成与文件名规则**属于 Application 职责：由 `finished_code / version` 生成 `sanitized_version` 并拼接目标路径；Domain 不得感知操作系统路径规则。

### 2.2 复用现有能力（关键）

- **Excel 库**：项目已使用 `ClosedXML` 进行导出，可复用其读取能力做 BOM 解析（只读）。
- **Normalize（公共模块）**：沿用 PRD `15.2`，BOM 审核对 `description` 必须走同一 Normalize（新增规则：`，`→`,`）。
- **物料创建用例**：BOM NEW 导入必须走既有“手工建料（Manual code）”语义的应用入口（或等价封装），确保唯一性/错误码一致，不另起一套“BOM 专用建料逻辑”。

## 3. 数据库设计与迁移（轻量）

### 3.1 新增表：`bom_archive`

表结构（与 PRD 对齐，不保存明细）：

```text
bom_archive
  id
  finished_code
  version
  file_path
  created_at
```

建议的额外约束（实现层面，便于保证“不覆盖”）：

- **唯一约束（强烈建议）**：`UNIQUE(finished_code, version)`
  - 用于强制“同成品编码 + 同版本禁止覆盖”的产品语义。

索引建议：

- `INDEX(finished_code, created_at DESC)`：用于历史列表按时间查询。

### 3.2 迁移策略

- 采用“幂等建表/建索引”策略：
  - 启动时或 schema ensure 阶段：若不存在 `bom_archive` 则创建；若不存在唯一索引则创建。
- 任何迁移失败不得破坏现有物料库表结构与数据（与 V1.3 数据安全风格一致）。

## 4. BOM Excel 解析方案（只读）

### 4.1 输入与输出对象（Application DTO）

输入：
- `file_path`（用户选择的本地路径）

输出（供 UI 展示）：
- `finished_code`（表头解析）
- `version`（表头解析）
- `rows[]`：每行包含
  - `status`（PASS/NEW/ERROR）
  - `code`、`name`、`spec`、`description`、`brand`
  - `error_reason`（仅 ERROR 有值，必须可直接展示）
- `first_error_row_no`（建议补充，定位辅助字段）
  - 说明：返回首个异常所在 **Excel 原始行号**（按源表格行号，不是 rows 数组下标）
  - 示例：`第28行缺少物料编码`
- 统计：
  - `total_count`
  - `pass_count`
  - `new_count`
  - `error_count`
  - `missing_code_error_count`（缺编码异常数，PRD增量要求）

### 4.2 模板兼容策略

- 参考 `docs/template/BOM.xls` 作为“列名匹配源”。
- 列名识别策略：
  - 优先按列名精确匹配：`编码/名称/描述/规格/品牌`
  - 对大小写与空格做最小容错（trim），但 **不做语义猜测**（避免误识别）。
- 表头解析策略：
  - 以模板中约定位置/键名提取 `成品编码/PCBA版本号`。
  - 若缺失则视为“文件格式不合法”，阻断进入审核（返回 UI 全局错误）。

### 4.3 不修改原文件（实现硬约束）

- 解析使用 ClosedXML **只读加载**（不调用 SaveAs）。
- 归档保存使用 **文件级 copy**（`File.Copy`）保存原文件二进制内容，不做重写。

## 5. 审核规则实现（Domain 规则 + Application 编排）

### 5.1 状态定义（以 PRD 增量修订为准）

对每一条 BOM 明细行（行对象包含 `code/spec/...`），按以下判定：

**PASS**
- `code` 非空
- `code` 在物料库存在
- `spec` 在物料库存在
- `code/spec` 对应一致（库内该 `code` 的 `spec` == BOM 行 `spec`）

**NEW（可导入）**
- `code` 非空
- `code` 不存在
- `spec` 在 **启用态(status=1)** 数据集中不存在（以 PRD `10.1`：同分类启用态唯一为准；废弃 spec 不占用唯一性）

**ERROR（仅展示）**：分型并输出 `error_reason`
- A：`code` 为空 → “缺少物料编码”
- B：`code` 存在 & `spec` 已存在 & 对应关系不一致 → “编码与规格冲突”
- C：`code` 存在 & `spec` 不一致 → “编码已存在但规格不一致”
- D：`code` 不存在 & `spec` 已存在 → “规格已存在，疑似重复建料”

> 关键：**V1.4 不启用 BOM 场景自动编码建料**。因此 `code` 为空的行永远不会成为 NEW，也不允许通过导入在本模块补齐编码。

### 5.2 与物料库比对的查询需求（Repository）

为避免 N+1 查询与性能抖动，建议在 Application 层进行批量查询编排：

- 批量按 `code` 查询：返回 `code -> (spec, status, category_code, ...)`
- 批量按 `spec` 查询（启用态 `status=1`）：返回 `spec -> (code, category_code, ...)`

实现要点：
- 仍然保持 Repository 只负责 SQL，不做规则判断。
- Application 把结果映射到 Domain 审核器做判定。
 - 单次分析优先批量查询 `code/spec`，避免逐行 N+1 查询。

## 6. NEW 导入（入库）方案

### 6.1 导入入口与分层落点

新增 Application 用例（示意命名）：

- `AnalyzeBomUseCase`：解析 Excel + 审核 + 输出行状态与统计（不落库、不归档）
- `ImportBomNewMaterialsUseCase`：
  - 支持 **单行导入** 与 **一键导入全部 NEW**
  - 导入后触发重新审核（刷新行状态与统计）

### 6.2 导入必须复用“Manual code 建料”语义

原因：
- BOM 已提供 `code`，且 PRD 明确本版本不自动生成编码。
- 物料库的唯一性、错误码与并发语义已在 V1.3 定稿，应复用同一路径，避免出现“BOM 专用插入”导致规则漂移。

落地策略：
- 对每条 `NEW` 行，调用既有“手工输入 code 的创建主物料（A）”等价用例（或封装后的内部方法）：
  - 输入：`code/spec/description/brand`（`name` 按现有规则由分类快照生成或与既有入参保持一致）
  - `description` 先调用公共 Normalize（含 `，`→`,`）
- 若导入失败（例如 `CODE_DUPLICATE` / `SPEC_DUPLICATE` / `CODE_FORMAT_INVALID` 等）：
  - 该行转为 `ERROR` 并填充 `error_reason`（映射为用户可读文案）
  - 重新跑一次审核输出整体状态

## 7. 归档保存（文件 + 索引）

### 7.1 允许归档的条件（与 PRD 一致）

仅当：
- `ERROR = 0`
- `NEW = 0`

才显示/启用按钮：`保存审核通过BOM`

### 7.2 保存路径规则

目标路径（PRD）：

```text
程序执行目录/BOM/{成品编码}/{PCBA版本}.xls
```

文件名与路径策略（V1.4 修订后采用，必须一致）：

- 数据库字段 `bom_archive.version`：保存 **原始版本号**（完整值，不修改，用于 UI 展示与检索）
- 文件系统使用 `sanitized_version` 作为文件名：
  - 将 Windows 非法文件名字符：`\ / : * ? " < > |`
  - 统一替换为：`_`
- 目标文件名示例：
  - 原始 `version = KC001/V1:2026`
  - `sanitized_version = KC001_V1_2026`
  - 归档文件名：`KC001_V1_2026.xls`

必须明确：

- **UI 展示使用原始 `version`**
- **文件系统路径使用 `sanitized_version`**
- `bom_archive.file_path` 保存真实归档路径（包含 sanitized 后的文件名）

### 7.3 重复版本（禁止覆盖）

实现层必须做到：
- 写入 `bom_archive` 前先检查 `(finished_code, version)` 是否已存在（或依赖唯一约束捕获冲突）
- 若存在：阻断保存，提示“版本已存在，禁止覆盖”

### 7.4 事务与原子性建议

归档保存涉及“文件 copy + DB insert”两个介质，建议采用如下顺序保证可恢复：

1) 先进行**目标路径冲突检查**
2) copy 文件到临时路径（同目录下 `*.tmp`）
3) 文件 copy 成功后，再写入 `bom_archive`（DB 事务）
4) DB 写入成功后，将临时文件 **原子重命名**为最终文件名
5) 任一步失败：
   - DB 失败：删除临时文件（不影响既有归档）
   - 文件重命名失败：回滚 DB（或标记记录无效并清理），避免索引指向不存在文件

> 该流程与 V1.3“恢复不得破坏当前数据”的工程风格一致：要么完全成功，要么不留下半成品。

临时文件清理（建议，避免半成品污染）：

- 归档失败时必须删除临时文件（同目录 `*.tmp`），不保留半成品。

## 8. WPF 页面与交互（MVVM）

### 8.1 新增页面：BOM审核中心

建议 UI 组成：

- 顶部：文件选择（按钮 + 路径显示）
- 表头区：成品编码、版本号
- 统计区：总数 / PASS / NEW / ERROR / **异常物料（缺编码）**
- 列表列（PRD增量要求）：
  - `状态 | 编码 | 名称 | 规格 | 异常原因 | 操作`
- 底部按钮：
  - `一键导入全部新料`（存在 NEW 时可用）
  - `保存审核通过BOM`（NEW=0 且 ERROR=0 时可用）

### 8.2 列表操作规则（必须）

- `NEW` 行：显示“导入物料库”
- `ERROR` 行：操作列为空（无导入按钮）
- `code 为空`：操作列为空（且异常原因固定为“缺少物料编码”）

### 8.3 ViewModel 状态机建议

拆分三个互不混用的状态：

- 文件解析/审核态：`Idle / Parsing / Ready / Failed`
- 导入态：`Idle / Importing / Imported / Failed`
- 归档态：`Idle / Archiving / Archived / Failed`

避免把“解析失败”与“导入失败”混成一个枚举导致 UI 逻辑互踩（参照现有 V1.3 的 UI 语义风格）。

## 9. 错误码与提示（建议清单）

建议新增（或复用既有）错误码，Application 统一返回给 UI：

- `BOM_FILE_INVALID`：文件不符合模板/缺关键列/缺表头字段
- `BOM_HEADER_MISSING`：成品编码或版本号缺失
- `BOM_FILE_LOCKED`：文件被占用（Excel 正在打开等），导致读取失败或归档 copy 失败
- `BOM_ARCHIVE_VERSION_EXISTS`：版本已存在禁止覆盖
- 导入相关复用既有物料错误码：`CODE_FORMAT_INVALID`、`CODE_DUPLICATE`、`SPEC_DUPLICATE`、`CATEGORY_NOT_FOUND`、`SUFFIX_INVALID` 等

提示映射（示例）：
- `BOM_FILE_INVALID`：全局弹窗/顶部提示
- `BOM_FILE_LOCKED`：提示“文件被占用，请关闭 Excel 后重试”（适用：读取失败、归档 copy 失败）
- `BOM_ARCHIVE_VERSION_EXISTS`：全局提示 + 引导用户更换版本
- `CODE_DUPLICATE/SPEC_DUPLICATE`：导入动作返回后将该行转 ERROR，并写入“异常原因”

## 10. 测试清单（可验证点）

### 10.1 Domain 单测（纯规则）

- `code 为空` → `ERROR(A)`，原因“缺少物料编码”
- `code/spec` 均存在且一致 → `PASS`
- `code 非空` 且 `code/spec` 均不存在 → `NEW`
- `code 不存在` 且 `spec 已存在` → `ERROR(D)`

### 10.2 Application 集成测试（SQLite + 关闭 UI）

- 解析模板成功：列识别、表头提取、行数统计正确
- `一键导入全部新料`：
  - 导入后 NEW 变 0；若发生重复则对应行变 ERROR，并刷新统计
- `保存审核通过BOM`：
  - 仅在 NEW=0 且 ERROR=0 可调用
  - 成功后：
    - 文件存在于目标目录
    - `bom_archive` 有对应记录
  - 重复版本：阻断且不覆盖

### 10.3 文件“原封不动”验证（关键）

- 归档前后对比文件哈希（或字节长度 + hash）：确保 copy 的是原文件，不是重写后的 Excel。
