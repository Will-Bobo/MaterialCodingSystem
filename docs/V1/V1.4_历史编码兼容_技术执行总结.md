# V1.4（历史编码兼容版）技术执行总结（落地实现）

更新时间：2026-04-10  
范围：V1.4 历史编码兼容（`start_serial_no` + Auto/ManualExistingCode 双模式建料）在现有仓库中落地，实现与测试全绿。

---

## 1. 执行目标与约束回顾

### 1.1 目标
- **支持两种建料方式**：
  - **Auto（默认）**：自动生成主物料编码（suffix=A）
  - **ManualExistingCode**：输入已有历史物料编码（suffix=A-Z）
- **Category 层新增 `start_serial_no`**：避免新系统占用历史编号区间；并对 Auto 生成规则做升级。
- **warning + confirm**：当手工输入的 `serial_no > start_serial_no` 时，触发强提示并允许确认后继续。

### 1.2 强约束（执行期遵守）
- 保持 **Clean Architecture + MVVM** 依赖方向：Presentation → Application → Domain → Infrastructure
- **持久化与查询维度统一为 `category_id`**（`category_code` 仅用于输入解析/UI 展示/导出）
- 并发重试策略：**仅 Auto** 模式允许事务级重试；Manual 指定编码冲突 **直接失败**
- 以“最小可落地”为原则：不引入新架构层、不引入复杂配置/调度系统

## 2. 关键行为变更（对外语义）

### 2.1 Category：新增起始流水号
- `category.start_serial_no`：必填、>=1、默认 1

### 2.2 Auto（自动建料）流水号升级规则
Auto 生成主料（suffix=A）流水号规则升级为：

\[
next\_serial\_no = \max(maxSerialNo,\ start\_serial\_no - 1) + 1
\]

- 首次生成不会小于 `start_serial_no`
- 若已存在更大的流水号，则继续递增
- **自动流水号永不回退**

### 2.3 ManualExistingCode（历史编码录入）
输入已有编码，例如：`ZDA0000123B`，系统解析：
- `category_code = ZDA`
- `serial_no = 123`
- `suffix = B`

校验与写入语义（最小可实施）：
- 格式合法（Trim + Uppercase；末位 suffix A-Z；中间 7 位数字流水号）
- `category_code` 必须与当前选择分类一致
- `serial_no >= 1`
- **group 解析唯一依据**：`category_id + serial_no`
- **创建权限**：
  - suffix == 'A'：若 group 不存在 → 允许创建 group + 插入 item(A)
  - suffix != 'A'：**group 必须存在**，否则失败（不允许隐式创建 group）
- 若 `serial_no > start_serial_no`：
  - Application 返回 `RequiresConfirmation=true`（warning）
  - UI 二次确认后以 `ForceConfirm=true` 再次提交继续执行

## 3. 代码落点与实现说明（按层）

### 3.1 Infrastructure / Schema（SQLite）
#### 3.1.1 DDL 更新
- `category` 增加列：
  - `start_serial_no INTEGER NOT NULL DEFAULT 1`

#### 3.1.2 最小迁移（无需额外 Migration 框架）
在应用启动建库逻辑中，增加“存在性检测 + ALTER TABLE”：
- 若旧库 `category` 缺失 `start_serial_no` 列：执行 `ALTER TABLE ... ADD COLUMN ... DEFAULT 1`

### 3.2 Infrastructure / Repository（查询维度收敛）
核心点：
- `GetMaxSerialNoAsync` 改为以 `category_id` 维度取 max（`category_code` 仅用于解析到 id）
- 新增 group 解析方法：`GetGroupIdByCategoryIdAndSerialNoAsync(categoryId, serialNo)`

### 3.3 Domain（解析/校验的纯逻辑）
新增历史编码解析器 `MaterialCodeParser`：
- Normalize：Trim + Uppercase
- Parse：`{category}{7位流水}{suffix(A-Z)}`
- 解析失败抛 `DomainException("VALIDATION_ERROR", ...)`

### 3.4 Application（用例编排与事务/重试边界）
新增统一入口：
- `CreateMaterialRequest`
  - `CodeMode: Auto | ManualExistingCode`
  - `ExistingCode`（manual 必填）
  - `ForceConfirm`（二次确认提交）
- `CreateMaterialResponse`
  - 支持 `RequiresConfirmation / WarningCode / Message`（warning + confirm 契约）

兼容入口保留：
- `CreateMaterialItemA(CreateMaterialItemARequest)`：继续可用，内部转调 `CreateMaterial(Auto)`

并发策略：
- Auto：沿用现有事务级重试（最多 3 次，基于 UNIQUE 冲突）
- Manual：不做“改号重试”，冲突直接 `VALIDATION_ERROR`

### 3.5 Presentation（WPF / MVVM）
在“新建主物料(A)”页新增：
- 编码方式单选：
  - 自动生成（默认）
  - 输入已有物料编码（ManualExistingCode）
- Manual 模式显示历史编码输入框 `ExistingCode`
- warning + confirm：
  - `RequiresConfirmation=true` 时弹窗确认
  - 确认后二次提交（`ForceConfirm=true`）
- 防重复提交：
  - `IsSubmitting` 状态控制按钮禁用（避免双击重复创建）

## 6. 风险与注意事项（评审重点）

### 6.1 Schema 最小迁移的覆盖面
- 当前采用 `SqliteSchema.EnsureCreated` 内的“列存在性检测 + ALTER TABLE”方式迁移
- 测试库 DDL 已同步增加 `start_serial_no`，避免集成测试出现缺列

### 6.2 ManualExistingCode 的写入一致性
- group 解析已统一为 `category_id + serial_no`
- suffix != A 的写入路径已阻断 “group 不存在自动创建”

### 6.3 warning + confirm 的实现方式
- 当前采用“**UI 二次确认 + 二次提交**”的实现契约
- `IsSubmitting` 用于按钮禁用，减少双击重复提交风险
