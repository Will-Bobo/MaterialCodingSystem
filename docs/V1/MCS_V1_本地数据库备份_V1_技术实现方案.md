# 技术方案文档（V1.2）——手动导出 + 自动备份 + 数据恢复（在 V1.1 基础上工程级优化补充）

> 本文为在 V1.1“修正与收敛”基础上的**工程级优化补充**（V1.2）。  
> 约束：不重新设计、不引入新复杂机制，仅补齐稳定性与安全性细节。  
> **保留不变**：`VACUUM INTO`（禁止 `File.Copy` 主库）、恢复=文件替换（禁止数据导入）、启动时自动备份一次、Cleanup保留20份、自动备份失败不影响主流程。

---

## 一、变更总览

### 1.1 已包含的修正点（V1.1）

- **修正点#1（最高优先级）**：主数据库路径唯一化：**`%LocalAppData%\MaterialCodingSystem\mcs.db`**（系统唯一主库路径）。
- **修正点#2**：Restore 成功后必须立即退出当前进程，并由 Presentation 触发重启（避免使用失效 Singleton 连接）。
- **修正点#3**：恢复安全校验：禁止恢复当前主库；最小文件合法性校验（`.db` + 可被 SQLite 打开）。
- **修正点#6（架构补充）**：新增 `IDatabasePathProvider`，所有路径统一由该接口提供，禁止硬编码/多来源。

### 1.2 新增的工程级优化点（V1.2，本次新增）

- **优化点#1（强制）**：Restore 操作全局互斥控制（保护 Export/AutoBackup/Restore 三个运维操作互斥执行）。
- **优化点#2（强制）**：`VACUUM INTO` 目标路径安全处理（禁止直接拼接未处理路径进入 SQL，避免引号等特殊字符导致注入/语义错误）。
- **优化点#3（可选）**：自动备份去重（当天已有自动备份可跳过）。
- **优化点#4（可选）**：Restore 用户提示增强（显示当前库时间与待恢复库时间，降低误操作）。
- **优化点#5（推荐）**：默认保留最近一次 `mcs.db.bak` 作为兜底回滚文件。

## 二、系统唯一路径（强制修正）

### 2.1 系统唯一主数据库路径（唯一来源）

**唯一主库路径必须统一为：**

- **主库**：`%LocalAppData%\MaterialCodingSystem\mcs.db`

明确声明（强制）：

- 这是系统**唯一主数据库路径**。
- **导出 / 自动备份 / 恢复** 任何操作均必须基于该路径。
- **不允许出现第二个来源路径**（例如 `AppContext.BaseDirectory/.../mcs.db` 之类路径在 V1.1 方案中视为不合规）。

### 2.2 备份目录（保持不变）

- **备份目录**：`%LocalAppData%\MaterialCodingSystem\backup\`

## 三、统一设计（保持原有正确设计）

### 3.1 统一机制：手动导出与自动备份共用同一导出能力

- **统一导出能力**：使用 SQLite **`VACUUM INTO`** 生成一致性快照文件。
- 差异仅在**目标路径**：
  - 手动导出：用户选择路径
  - 自动备份：固定备份目录 + 标准命名 + Cleanup

### 3.2 禁止项（保持不变）

- 禁止 `File.Copy` 直接复制主库作为备份手段（风险：不一致快照）。
- 禁止“数据导入”方式恢复（逐表导入/SQL回灌），恢复只能做文件替换。

## 四、功能定义（V1.1 口径）

### 4.1 `ExportDatabase(path)`

- **输入**：用户选择的目标文件路径 `path`
- **行为**：对“唯一主库”执行 `VACUUM INTO path`
- **备注**：不做保留策略（用户自行管理导出文件）

### 4.2 `CreateAutoBackup()`

- **输入**：无（系统触发）
- **行为**：
  - 生成备份文件名 `mcs_yyyyMMdd_HHmmss.db`
  - 导出到 `%LocalAppData%\MaterialCodingSystem\backup\`
  - Cleanup：保留最近20份
- **失败策略（保持不变）**：失败不影响主流程，仅日志

### 4.3 `RestoreDatabase(path)`

- **输入**：用户选择的源数据库文件路径 `path`
- **行为**：
  - 执行恢复前校验（见第六章）
  - 关闭数据库连接
  - 原子替换 `mcs.db`
  - **恢复成功后必须立即退出当前进程，并由 Presentation 触发重启**

## 五、分层设计（在原方案基础上补齐路径提供者）

### 5.1 Application：`DatabaseBackupService`（保持职责不变）

- 只编排流程、做参数校验、返回 `Result<T>`
- 通过 Infrastructure 接口执行 `VACUUM INTO`、文件替换与校验
- 通过 **`IDatabasePathProvider` 获取主库路径与备份目录**（修正点#6）

#### V1.2 新增：运维类操作互斥控制（优化点#1，强制）

必须补充说明：

- **Restore 执行期间必须保证没有其他数据库运维操作并发执行**（导出/自动备份/恢复三者互斥）。
- 明确口径：**“Restore 操作必须具备全局互斥控制，避免与 Export / AutoBackup 并发执行导致连接关闭失败或文件占用问题。”**

实现建议（写入方案，供后续开发）：

- 在 Application 层引入轻量级互斥（如 `SemaphoreSlim(1,1)`），仅保护：
  - `ExportDatabase`
  - `CreateAutoBackup`
  - `RestoreDatabase`
- 互斥**不影响正常业务查询/写入用例**（只保护“运维类操作”，避免扩大锁粒度导致业务卡顿）。

### 5.2 Infrastructure：`SqliteBackupRepository`（保持职责不变）

- 执行 `VACUUM INTO`
- 目录/文件操作（list/delete/replace）
- 连接关闭（为恢复释放文件句柄）
- 恢复前/后校验（打开SQLite、可选 integrity_check）

#### V1.2 新增：`VACUUM INTO` 路径安全处理（优化点#2，强制）

必须补充说明：

- 禁止将**未经处理的用户输入路径**直接拼接到 `VACUUM INTO '...';` SQL 中。
- 风险点：
  - 路径可能包含特殊字符（尤其是 **单引号 `'`**），会破坏 SQL 字符串字面量并形成注入/语义错误。

处理要求（满足其一即可，推荐优先级从高到低）：

- **推荐 A：安全转义**  
  - 在拼接到 SQL 字符串字面量前，对路径中的单引号进行 SQL 字符串转义：`'` → `''`，再执行：`VACUUM INTO '<escapedPath>';`
- **可选 B：严格校验 + 受控字符集**  
  - 明确限制目标路径必须为本地文件路径，拒绝包含控制字符等；并仍需处理引号等字符。

备注：

- `VACUUM INTO` 通常不支持常规参数化（即便支持也应以官方可行方式为准），因此 V1.2 方案要求在 Infrastructure 内实现**明确且可测试的转义/校验策略**，避免注入与执行失败。

### 5.3 Domain：不参与（保持不变）

- 理由不变：备份/恢复是基础设施可用性能力，不承载业务规则。

### 5.4 新增：`IDatabasePathProvider`（修正点#6，必须补充）

**职责：**

- `GetMainDbPath()`：返回唯一主库路径 `%LocalAppData%\MaterialCodingSystem\mcs.db`
- `GetBackupDirectory()`：返回备份目录 `%LocalAppData%\MaterialCodingSystem\backup\`

**强制约束：**

- 所有模块（Application / Infrastructure / Presentation 组合根）必须通过该接口获取路径
- 禁止硬编码路径、禁止从 BaseDirectory 等第二来源推导主库路径

## 六、恢复（Restore）修正与安全校验（强制新增）

### 6.1 禁止恢复当前主库（修正点#3）

- 若 `selectedPath == currentDbPath`：
  - **拒绝操作**
  - 返回失败（建议错误码：`DB_RESTORE_SOURCE_IS_CURRENT_DB`）

### 6.2 文件合法性最小校验（修正点#3）

最小要求（强制）：

- 必须是 `.db` 文件（扩展名校验）
- 必须能被 SQLite 打开（最基础校验）

说明：这不是“导入校验”，只是确保不是随便的文件/损坏文件，避免把系统置于不可用状态。

### 6.3 Restore 前文件有效性校验（优化点#5）

- **必须**：尝试打开 SQLite（例如 open connection + 简单查询/pragma）
- **可选**：`PRAGMA integrity_check;`
- 至少保证：不是损坏文件/非 SQLite 文件，否则拒绝恢复（建议错误码：`DB_RESTORE_SOURCE_INVALID` / `DB_RESTORE_INTEGRITY_CHECK_FAILED`）

## 七、Restore 后的进程生命周期（强制修正）

### 7.1 必须明确的事实（修正点#2）

- 当前系统 DI 中 `SqliteConnection` 是 **Singleton + 已打开**
- Restore 使用文件替换后：
  - 该连接对象要么已被关闭/Dispose
  - 要么其内部状态与文件句柄关系已不可依赖
  - **继续执行业务逻辑将使用失效连接，导致不可预测问题**

### 7.2 强制行为（修正点#2）

恢复成功后必须：

- **立即终止当前进程**
- **不允许继续执行任何业务逻辑**
- **必须由 Presentation 层触发应用重启**

必须在实现/交互中明确写出以下语句（评审冻结口径）：

- **“Restore 成功后必须立即退出当前进程，否则会使用失效连接，导致不可预测问题。”**

## 八、自动备份触发时机优化（建议纳入，保持“启动一次”不变）

### 8.1 触发时机（优化点#4）

将“启动立即执行”调整为以下二选一（均满足“启动时执行一次”）：

- **App 启动完成后**触发（例如 `MainWindow Loaded` 事件后）
- 或 **延迟 3~5 秒**触发（启动完成后调度一次）

### 8.2 原因说明（必须写入方案）

- 避免初始化阶段 DB 未稳定（首次建库/EnsureCreated/首次事务）
- 降低锁冲突概率（减少与启动期写事务/查询竞争）

### 8.3 失败策略（保持不变）

- 自动备份失败：仅日志，不弹窗，不阻断主流程。

#### V1.2 可选：自动备份去重（优化点#3，可选）

补充说明（可选，不强制）：

- 若当天已经存在自动备份文件，可选择跳过本次自动备份（避免重复文件与无意义占用）。
- 判定口径建议（不改变既有命名规则前提下）：
  - 以备份文件名日期前缀（`mcs_yyyyMMdd_*.db`）或文件创建时间落在当天进行判断。
- 即使启用去重：
  - **Cleanup 仍按保留20份执行**（保持不变）
  - 自动备份失败仍不影响主流程（保持不变）

## 九、文件替换原子性（保持原方案，强调“保护原 DB”）

- 恢复必须使用“临时文件 + 原子替换”的策略，避免半写主库：
  - 复制到 `mcs.db.restore.tmp`
  - 校验通过后再替换 `mcs.db`
  - 失败时保证 `mcs.db` 原样可用
#### V1.2 推荐：默认保留最近一次 `mcs.db.bak`（优化点#5，推荐）

- 建议将“保留旧库为 `mcs.db.bak`”从可选提升为**默认行为**：
  - 作为最后兜底回滚手段
  - Restore 异常或用户误操作时可快速恢复
- 约束：
  - 仅保留**最近一次**（不引入多版本回滚体系，避免复杂化）
  - `.bak` 仅作为本地兜底文件，不改变主库路径唯一性口径

## 十一、错误处理（保持原则，补充关键错误）

- 自动备份失败不影响主流程（保持不变）
- 恢复失败必须保护原 DB（保持不变）

建议补充错误码（示例）：

- `DB_RESTORE_SOURCE_IS_CURRENT_DB`：禁止选择主库自身恢复
- `DB_RESTORE_SOURCE_INVALID`：非 `.db` 或无法打开
- `DB_RESTORE_INTEGRITY_CHECK_FAILED`：完整性校验失败（如启用）
- `DB_RESTORE_REPLACE_FAILED`：原子替换失败
- `DB_RESTORE_RESTART_REQUIRED`：恢复成功（用于驱动 UI 立即退出并重启）

### V1.1 收敛结论

- **路径唯一化**是最高优先级：主库必须固定到 `%LocalAppData%\MaterialCodingSystem\mcs.db`，彻底消除多来源。
- **恢复后必须立即退出并重启**：这是对当前 Singleton 已打开连接模型的必然约束。
- **恢复校验最小闭环**：禁止选择主库自身 + `.db` + 可打开（可选 integrity_check），避免把系统恢复到不可用状态。
