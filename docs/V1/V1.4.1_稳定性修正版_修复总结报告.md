## 2. 关键修复点（实现摘要）

### 2.1 SQLite Migration 机制增强（schema_version）
- 新增 `app_meta(k,v)` 表管理 `schema_version`
- 启动（`SqliteSchema.EnsureCreated`）逻辑：
  - 若无 `app_meta`/`schema_version` → 初始化为 `1`
  - 若无 `category` 表（全新库）→ 直接创建最新 schema 并设置 `schema_version=3`
  - 若 `version < 当前版本` → 按版本顺序执行迁移（幂等）
  - 迁移后做 **drift 校验**：版本声称已升级但关键结构缺失 → **抛错阻断运行**

**版本路径**
- `1 -> 2`：补齐 `category.start_serial_no`（含数据修复：NULL → 1）
- `2 -> 3`：创建 `create_request_log`（幂等/幂等日志）

### 2.2 ManualExistingCode：group 已存在时后补 A
支持场景：
- 已存在 `ZDA0000123B`
- 后补录 `ZDA0000123A`

行为：
- 复用同一 `category_id + serial_no` 的 group
- 插入 A item 成功
- 若 A 已存在，再补 A：
  - 触发 `UNIQUE(group_id, suffix)` 或 `UNIQUE(code)`，映射为 `VALIDATION_ERROR`

### 2.3 CreateMaterial 幂等（RequestId）
在 `CreateMaterialRequest` 增加：
- `RequestId: string?`

success-only 幂等语义：
- 若 `RequestId` 为空：保持原逻辑
- 若 `RequestId` 有值：
  - 先查 `create_request_log` 是否已有成功结果
  - 若存在：直接返回首次成功结果
  - 若不存在：按正常流程创建；成功后写入 log
  - 并发重复写 log 触发唯一约束时：回查并返回已存在结果（不重复创建数据）

### 2.4 Parser：Longest Prefix Match（基于 DB 分类 code 集合）
手工输入历史编码解析前置步骤：
- 从 Repo 拉取 `category.code` 列表
- 在 `ExistingCode` 上做最长前缀匹配（且剩余长度必须为 `8`：7位流水 + suffix）
- 无法匹配 → `VALIDATION_ERROR`
- 匹配后再由 Domain 完成尾部解析（7位流水 + suffix A-Z）

## 5. 风险提示与上线建议

### 5.1 Migration 失败策略
- V1.4.1 起：迁移/校验失败会 **抛错阻断运行**（符合“禁止静默继续”要求）
- 建议上线前按 PRD 的 DB 备份能力先做一次全量备份（已实现）

### 5.2 幂等策略边界
- 当前为 **success-only 幂等**（最小实现）
- 若未来需要覆盖失败幂等、跨进程重试语义，可扩展 log 字段（不在本次范围）
